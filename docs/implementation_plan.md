# LSLM v4 実装フロー

## 0. 本書の目的

本ドキュメントは、LSLM v4 の思想・アーキテクチャ・実装安全性の 3 観点から、最小構成で破綻しにくい実装順序を定義する。

v4 の実装は、過去資産をそのまま復元することではない。  
**知識ネットワークとしての辞書を中核に、Plan / Divergence / Convergence / Slot / Surface の責務分離を成立させること** が目的である。

---

## 1. 前提

LSLM v4 は以下の 4 分離を厳守する。

- 辞書 = 知識ネットワーク
- 実行時 = 状態とワーキングメモリ
- 保存 = ログ / 学習データ / 評価結果
- 設定 = 実験条件 / しきい値 / LLM使用順序

この分離を崩さないことが最重要原則である。

また、辞書は次のように定義する。

> **辞書は、ただの知識ではなく、知識のネットワークそのものである。**

---

## 2. 実装優先順位の全体像

v4 の初期実装は、次の順序で進める。

1. スキーマ固定
2. 辞書仕様の最小完成
3. 辞書 I/O 基盤
4. ログ基盤の常設
5. Plan の最小実装
6. Divergence v1
7. Convergence v1
8. Slot v1
9. Surface v1
10. 最小縦スライス統合
11. 段階別 scoring
12. reward 導入
13. 外部 evaluator 接続
14. 辞書還元ループ整備

---

## 3. スキーマ固定

### 目的

全モジュールの入出力契約を先に確定し、後工程での破綻を防ぐ。

### 定義対象

- 辞書エントリ構造
- 実行時 State 構造
- InputFeatures
- IntentPlan
- DivergenceCandidate
- ConvergenceCandidate
- SlotFrame / FilledSlots
- SurfacePlan
- ResponseResult
- JSONL Trace Record
- Reward Record

### 完了条件

- 全構造が dataclass または JSON Schema で定義済み
- 各モジュール間の受け渡しが明文化されている

---

## 4. 辞書仕様の最小完成

### 目的

LSLM v4 の知識ネットワークを最小単位で成立させる。

### 必須要素

- Axis
- Grammar
- Slot
- Relation
- Meta / Index

### 完了条件

- 少数語彙で辞書ソースが成立する
- すべてのエントリが必要最小限の構造を持つ
- Relation が最低限ネットワークとして機能する

---

## 5. 辞書 I/O 基盤

### 目的

辞書を安定してロード・変換・参照できるようにする。

### 実装内容

- 辞書ソース → 実行用辞書形式への変換
- 実行用辞書ロード
- インデックス生成
- バリデーション
- メタ情報の自動補完

### 完了条件

- `src/core/io/lsd_lexicon.py` 相当のコアが安定動作する
- Axis / Grammar / Slot / Relation / Index を取得できる

---

## 6. ログ基盤の常設

### 目的

全思考プロセスを可視化し、後の学習・分析を可能にする。

### 記録対象

- input
- input_features
- plan
- divergence 候補
- convergence 候補
- slot 構造
- surface plan
- 最終出力
- スコア内訳
- 棄却理由
- relation 経路
- 所要時間

### 完了条件

- 1入力につき1トレースが JSONL で出力される
- 中間状態がすべて追跡可能

---

## 7. Plan v1

### 目的

入力から応答の骨子を決める。

### 初期対象

- 質問への応答
- 説明
- 確認
- 雑談応答

### 完了条件

- 入力に対して `intent` と `required_slots` が必ず得られる
- フォールバック時も最低限の plan を返せる

---

## 8. Divergence v1

### 目的

知識ネットワーク上で関連候補を広げる。

### 手法

- 入力語からの起点抽出
- Relation 展開
- Axis 近傍探索
- カテゴリ制約付き候補追加

### 完了条件

- 複数候補が安定して生成される
- 探索経路がログに残る

---

## 9. Convergence v1

### 目的

広げた候補から必要要素を選別する。

### 判定軸

- Plan 適合度
- 入力との関係性
- 冗長性の低さ
- 矛盾の少なさ
- コスト効率

### 完了条件

- 候補群から採用候補が選べる
- 棄却理由が説明可能

---

## 10. Slot v1

### 目的

意味構造を確定する。

### 対象

- actor
- target
- state
- time
- location
- cause
- conclusion

### 完了条件

- 必要 slot が埋まる
- 未充填 slot が明示される

---

## 11. Surface v1

### 目的

意味構造を自然な文章に変換する。

### 手法

- テンプレートベース
- 接続表現選択
- 文順調整
- 文体整形

### 制約

- 新しい意味を追加しない
- slot 構造外の内容を勝手に補わない

### 完了条件

- 文として成立する出力が生成される

---

## 12. 最小縦スライス統合

### 目的

システムを 1 本通して動作させる。

### フロー

```text
input
→ plan
→ divergence
→ convergence
→ slot
→ surface
→ response
```

### 制約

- intent は少数に限定
- 高度な分岐は禁止
- 外部LLM必須にしない

### 完了条件

- 任意入力で最低1応答が返る
- トレースが完全に出力される

---

## 13. 段階別 scoring

### 目的

各段階の品質を個別に評価できるようにする。

### 初期スコア要素

- plan fitness
- divergence relevance
- convergence fitness
- slot completeness
- surface naturalness
- input retention
- grammar fitness

### 完了条件

- 各段階のスコアがログへ記録される

---

## 14. reward 導入

### 目的

学習や候補選択に用いる統合報酬を導入する。

### 方針

- `reward.internal`
- `reward.external`
- `reward.total`

の 3 分離を採用する。

### 完了条件

- reward 内訳が 1 ターンごとに保存される

---

## 15. 外部 evaluator 接続

### 目的

人間視点に近い品質信号を補助的に取り込む。

### 対象

- Gemini
- OpenAI
- Local evaluator

### 注意

- 外部評価器は補助用途に限定する
- 中核推論を外部LLMへ丸投げしない
- 使用順序は `settings/LLM_order.yaml` で管理する

### 完了条件

- 最終応答に対する external score が取得できる

---

## 16. 辞書への還元

### 目的

知識ネットワークを安全に改善する。

### 対象

- relation 追加候補
- slot フレーム補強
- axis 微調整候補
- alias / category 補強

### 注意

- 自動反映は禁止
- 必ず検証を挟む
- 失敗例も保存する

### 完了条件

- 辞書更新候補が安全に提案される

---

## 17. 実装順序に関する注意

### 17.1 Surface を先に作りすぎない
見た目の自然さから入ると、中身の責務分離が崩れやすい。

### 17.2 外部LLM依存で縦スライスを完成扱いしない
外部LLMは補助であり、本体ではない。

### 17.3 辞書を単語帳に戻さない
辞書は知識ネットワークであり、単語属性表へ退化させてはならない。

### 17.4 ログを後回しにしない
ログがない実装は、v4 では未完成とみなす。

---

## 18. まとめ

LSLM v4 の実装は、

- まずスキーマを固定し
- 次に辞書ネットワークを成立させ
- その上に Plan / Divergence / Convergence / Slot / Surface を順に積み
- 最後に学習と評価を接続する

という順番が最も安全である。

特に、

> **辞書を知識ネットワークとして先に成立させること**

が、v4 全体の成否を決める最重要条件となる。
