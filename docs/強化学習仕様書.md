# LSLM v3 強化学習仕様書（reward.external 導入版）

## 0. 目的

本仕様書は、LSLM v3 における強化学習系の報酬設計を定義するものである。
特に本版では、外部評価器を接続し、`reward.external` を実値で埋める段階に入ったことを前提に、

* `reward.internal`
* `reward.external`
* `reward.total`

の3分離を中核仕様として定義する。

本仕様の目的は以下である。

* LSLM v3 の思想に整合した報酬設計を確立する
* 外部評価器導入時の責務分離を明確化する
* 学習・ログ・分析で扱うデータ構造を固定する
* 今後の多軸 external 評価への拡張余地を残したまま、現段階で最も合理的な第一実装を定義する

---

## 1. 前提思想

LSLM v3 は、単一のブラックボックス報酬へ依存する設計ではない。
辞書知識・実行時状態・保存記録を分離し、さらに推論も

* Intent Planning
* Semantic Recall
* Slot Filling
* Surface Realization
* Scoring

へ分割して扱う。

この思想に従う場合、報酬もまた一括スカラーとしてのみ扱うのではなく、

* 内部構造的な正しさ
* 外部評価器による対話品質評価
* 学習更新に用いる統合値

に分けて保持するほうが合理的である。

したがって本仕様では、報酬を以下の3要素に分離する。

```python
reward = {
    "internal": 0.0,
    "external": 0.0,
    "total": 0.0,
}
```

---

## 2. 報酬設計の結論

現段階における最適方針は以下とする。

1. `reward.internal` は LSLM v3 内部の構造的・規則的評価を表す
2. `reward.external` は evaluator の実値評価を表す
3. `reward.total` は両者を統合した最終学習報酬とする
4. 学習ログには3値すべてを保存する
5. 現段階では `reward.external` は単一スカラーで扱う
6. 将来的には `reward.external` を複数軸へ分解可能な構造に拡張する

---

## 3. 採用理由

### 3.1 思想整合性

LSLM v3 は、v2 における「学習が一括報酬に依存していた」問題を解消する方向で再設計されている。
そのため、報酬を分解し、内部整合性と外部品質を分けて扱うことは思想上自然である。

### 3.2 学習工学上の合理性

外部評価器は高表現力だがノイズを含む。
一方、内部評価は低ノイズで安定しているが、人間視点の自然さや有用性を十分には担保しない。

このため、

* `internal` = 安定だが限定的
* `external` = 強力だが揺れる

という性質を組み合わせることが、学習初期段階では最も合理的である。

### 3.3 運用・デバッグ性

3分離により、以下の切り分けが可能になる。

* `internal` が低いのか
* `external` が低いのか
* 両方低いのか
* `internal` は高いが `external` が低いのか

これにより、失敗原因の観測可能性が大きく向上する。

---

## 4. reward.internal の定義

## 4.1 役割

`reward.internal` は、LSLM v3 自身の内部構造がどれだけ健全であるかを評価する報酬である。

これは「思考エンジンとしての正しさ」を担保する基礎報酬であり、外部評価器の有無に関わらず算出される。

## 4.2 構成要素

現段階では最低限以下を用いる。

* semantic_consistency
* slot_fitness
* grammar_fitness
* input_retention
* policy_fitness（必要に応じて）

## 4.3 基本式

```python
reward.internal = (
    w_semantic * semantic_consistency +
    w_slot     * slot_fitness +
    w_grammar  * grammar_fitness +
    w_input    * input_retention +
    w_policy   * policy_fitness
)
```

`w_policy` は未導入フェーズでは 0.0 でもよい。

## 4.4 正規化

`reward.internal` は 0.0〜1.0 に正規化する。

```python
reward.internal = max(0.0, min(1.0, reward.internal))
```

---

## 5. reward.external の定義

## 5.1 役割

`reward.external` は、外部評価器が最終応答を評価した結果を実値として取り込む報酬である。

これは「人間から見た品質」に近い信号を導入するための報酬であり、

* 自然さ
* 有用性
* 一貫性
* 応答としての妥当性

などを外部から与える役割を持つ。

## 5.2 evaluator の前提

利用可能 evaluator は以下を想定する。

* Gemini
* OpenAI
* ローカル LLM

初期段階では、いずれか1系統でよい。
複数 evaluator の比較やアンサンブルは後段階とする。

## 5.3 入力

evaluator へ渡す最低限の材料は以下とする。

* user_input
* inferred_intent
* filled_slots（必要に応じて要約）
* final_response
* 補助メタデータ（任意）

ただし、初期段階では evaluator に過剰な中間状態を渡しすぎない。
まずは最終応答品質の評価器として導入する。

## 5.4 出力

現段階では evaluator 出力を以下に正規化する。

```python
reward.external: float  # 0.0 ~ 1.0
feedback_text: str
label: str
```

## 5.5 正規化

```python
reward.external = max(0.0, min(1.0, evaluator_score))
```

必要に応じて、良回答を強調するため次のような非線形変換を許可する。

```python
reward.external = reward.external ** 1.5
```

ただし、これは設定で有効化できるオプションとし、初期値は無効でもよい。

## 5.6 現段階での制約

現段階では `reward.external` を単一スカラーで扱う。
理由は以下である。

* 外部評価導入の初期実装を簡潔に保てる
* ログと学習パイプラインの接続を先に安定化できる
* 多軸 external は後から拡張可能である

---

## 6. reward.total の定義

## 6.1 役割

`reward.total` は、学習更新や候補選択に用いる統合報酬である。

これは `internal` と `external` の両方を反映する唯一の最終値であり、保存対象でもある。

## 6.2 基本式

```python
reward.total = (
    alpha * reward.internal +
    beta  * reward.external
)
```

制約:

```python
alpha + beta = 1.0
```

## 6.3 推奨初期値

導入初期は以下を推奨する。

```python
alpha = 0.8
beta = 0.2
```

理由:

* evaluator ノイズの影響を抑える
* 内部構造を先に安定させる
* 外部評価器を補助信号として導入する

## 6.4 フェーズ別推奨比率

### 初期

```python
reward.total = 0.8 * internal + 0.2 * external
```

### 中期

```python
reward.total = 0.6 * internal + 0.4 * external
```

### 後期

```python
reward.total = 0.3 * internal + 0.7 * external
```

ただし後期比率へ移行する条件は、

* evaluator の安定性が確認されている
* internal が十分成熟している
* ログから external の有用性が検証されている

場合に限る。

---

## 7. external 導入時の禁止事項

以下は禁止とする。

### 7.1 external のみで学習すること

初期段階で `reward.external` 単独へ依存してはならない。
理由は evaluator ノイズと設計逸脱リスクが高いためである。

### 7.2 internal を削除すること

LSLM v3 の内部構造的健全性は、引き続き内部報酬で追跡しなければならない。

### 7.3 evaluator の自由文だけを保存して数値を保存しないこと

フィードバック文だけでは後から統計処理・再学習・比較分析が困難になる。
必ず数値スコアを保存する。

### 7.4 total のみを保存して internal / external を捨てること

原因分析不能になるため禁止。

---

## 8. ログ仕様

## 8.1 保存方針

`reward.internal` / `reward.external` / `reward.total` は必ずログ保存対象とする。

## 8.2 最低保存項目

```json
{
  "reward": {
    "internal": 0.0,
    "external": 0.0,
    "total": 0.0
  },
  "evaluation": [
    {
      "evaluator_name": "gemini",
      "score": 0.0,
      "label": "",
      "feedback": "",
      "metadata": {}
    }
  ]
}
```

## 8.3 併せて残すべきもの

* input
* inferred intent
* slot 構造
* candidate 群
* candidate ごとの score breakdown
* chosen response
* evaluator feedback
* 使用モデル名
* フォールバック有無
* エラー情報（失敗時）

## 8.4 目的

これにより、

* internal 高 / external 低
* internal 低 / external 高
* 両方低
* 両方高

を後から分析できる。

## 10. evaluator 接続仕様

## 10.1 インターフェース

```python
class ExternalEvaluator:
    def evaluate(
        self,
        user_input: str,
        final_response: str,
        context: dict | None = None,
    ) -> EvaluationResult:
        ...
```

## 10.2 実装要件

* evaluator 実装は差し替え可能にする
* Gemini / OpenAI / Local を共通 I/F で包む
* 失敗時は例外を握り潰さずログへ残す
* 評価失敗時は `reward.external = 0.0` 固定ではなく、明示的に失敗状態を保存する

## 10.3 評価失敗時の方針

評価失敗時は以下のどちらかを採用する。

### 推奨方式A

```python
reward.external = previous_default_external
```

たとえば `0.5` などの中立値。

### 方式B

```python
reward.external = reward.internal
```

内部評価へ退避する方式。

ただしどちらを使ったかは必ずログに残す。

---

## 11. 学習更新仕様

## 11.1 初期段階の原則

学習更新は `reward.total` を使って行う。
ただし分析用途として `internal` と `external` は常に別保存する。

## 11.2 候補比較への利用

候補生成が複数ある場合、

* まず `reward.internal` で粗い足切り
* 次に `reward.total` で最終選択

としてよい。

これにより evaluator 呼び出し回数の節約が可能になる。

## 11.3 将来拡張

将来的には以下を許可する。

* internal を段階別報酬へ分解
* external を複数軸へ分解
* total を信頼度付き合成へ変更

---

## 12. 将来拡張仕様

現段階では `reward.external` は単一スカラーだが、次段階では以下への拡張を想定する。

```python
external = {
    "naturalness": 0.0,
    "helpfulness": 0.0,
    "consistency": 0.0,
    "safety": 0.0,
    "total": 0.0,
}
```

この多軸化は、

* evaluator が安定稼働している
* ログ基盤が安定している
* 単一 external では改善方向が粗い

と確認された段階で導入する。

---

## 13. 段階的導入手順

### Phase 1

* `reward.internal` を既存通り維持
* evaluator を1系統接続
* `reward.external` を単一スカラーで保存
* `reward.total = 0.8 * internal + 0.2 * external`

### Phase 2

* evaluator の失敗時処理を安定化
* フィードバック文の保存を追加
* モデル名・フォールバック情報を完全記録

### Phase 3

* external 比率を 0.4 以上へ引き上げ検証
* evaluator の一致率分析を実施
* 候補比較への external 利用を拡大

### Phase 4

* external の多軸化
* 信頼度付き統合報酬へ移行

---

## 14. 最終結論

本仕様では、LSLM v3 における強化学習報酬を

* `reward.internal`
* `reward.external`
* `reward.total`

の3分離で定義する。

この方針は、

* 思想整合性
* 学習工学上の合理性
* 運用・デバッグ性

の3観点から、現段階で最も合理的な第一解である。

なお、本仕様は完成形ではなく、

* 現在は単一 external
* 将来は多軸 external
* 最終的には信頼度付き統合

へ段階的に拡張される前提の暫定最適仕様である。

しかし、外部評価導入フェーズにおいては、この3分離仕様が最も破綻しにくく、最も観測可能性が高く、LSLM v3 の設計思想にも最も整合する。
